QubesOS, how to not get hacked when developing

2026-06-22

introduction

developing means running a lot of code i did not write: dependencies, build scripts, containers, npm packages, python packages. with the ongoing supply chain attacks in 2026 this became more and more troublesome. even with a yubikey and an encrypted password store, running untrusted code became a very serious risk. so i decided i had to do something about it.

QubesOS

after hearing a lot of good things about QubesOS, and seeing the ongoing attacks, i finally had the motivation to jump in and improve my security. i am currently running the i3 window manager and the archlinux community template.

my setup

  • personal for normal browsing, chat and memes
  • work for writing code and using ai agents
  • vault where the yubikey is connected for ssh and gpg keys
  • ssh only for ssh connections, firewalled to the ssh port, using Qubes.SshAgent policies from vault
  • pass for password storage, firewalled to the git repo, using Qubes.GpgAgent policies from vault
  • git for cloning, signing and committing, using Qubes.Git policies for the work qube, firewalled to github
  • disposable qubes for opening random files and links

the important part is that the risky stuff does not live next to the valuable stuff. if a build script does something bad, it should only hurt the qube where i ran it.

development workflow

i keep repositories in a git qube and clone them into the work qube with qubes-git-remote. gpg and ssh keys stay in the vault and are not just sitting in every environment where code can execute.

it is not magic and it does not remove the need to think. but it changes the failure mode. one mistake should not immediately mean that my whole machine, browser session, keys and projects are gone.

conclusion

QubesOS is slower and more work than a normal linux desktop, but for development it is worth it to me. it gives me a clear place to put risk, and that makes working on unknown code feel sane again.
